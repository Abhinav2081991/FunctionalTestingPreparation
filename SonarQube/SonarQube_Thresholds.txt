What Does “Threshold” Mean in SonarQube?

A threshold defines:

- When a rule should raise an issue
- How severe that issue is
- When it should fail the Quality Gate

Bad thresholds = noisy Sonar → ignored by teams

Good thresholds = trusted Sonar → enforced in CI

3️⃣ findElement Outside POM → MAJOR FAIL

🚨 This cannot be done reliably with regex/XPath

Why?

Sonar must know:

Is this class a Page Object?

Is findElement() used inside test or step files?

✅ Correct Solution: Custom Java Rule (Plugin)
Logic (AST-based)
IF methodInvocation.name == "findElement"
AND class does NOT:
   - extend BasePage
   - OR belong to *.pages.* package
THEN raise issue

Tools Needed

sonar-java-plugin

JavaParser / Sonar Java API

Severity
Setting	Value
Severity	Major
Gate	Fail

💡 This is a strong interview-worthy customization.

4️⃣ Assertions in Page Classes → MAJOR FAIL
XPath-based (Good enough)
XPath
//METHOD_INVOCATION[
 contains(@methodName,'assert')
 or contains(@methodName,'Assert')
]

Scope Limitation

Apply only to:

**/pages/**

Why it works

Assertions (assertTrue, Assertions.assertThat)
Should only be in test layers

5️⃣ Missing Wait Strategy → CRITICAL FAIL

🚨 Again — needs AST analysis

Rule Definition

ExpectedConditions

Custom WaitStrategy

→ Raise issue

Pseudocode

   reportIssue();
}

Why XPath fails

You must correlate method calls, not just match text.
