1️⃣ The 3 Ways to “Add Rules” in SonarQube
Level								Can we do it?			Typical Use Case
Enable existing rules				✅ Yes					Most common
Create custom rules via plugins		✅ Yes					Enterprise / advanced
Import rules from external tools	✅ Yes					Checkstyle, PMD, SpotBugs

⚠️ You cannot write new rules directly from the Sonar UI (except templates).

2️⃣ Option 1 – Enable / Disable Existing Rules (Most Common)

SonarQube ships with thousands of built-in rules.

How:

SonarQube → Quality Profiles →  Select language (Java) →  Activate / deactivate rules  → Assign profile to project

✔ No coding
✔ Fast
✔ 90% of real projects use this

Example (Automation Framework)

Enable rules like:
❌ Thread.sleep() usage
❌ Empty catch blocks
❌ Hard-coded waits
❌ Static WebDriver usage

3️⃣ Option 2 – Create Custom Rules (Advanced / Enterprise)

Used when:

Company has internal coding standards
Automation-specific violations
Architecture rules

Two supported approaches:

🔹 A) Custom Java Rule (Plugin)

You write a SonarQube plugin.
Steps (High-level):

Create a Maven project
Use Sonar Java Plugin APIs
Write AST-based rule
Package as JAR
Drop into extensions/plugins
Restart SonarQube

Example Custom Rule

“Fail if Selenium test uses Thread.sleep() instead of WaitStrategy”

		@Rule(
		  key = "NoThreadSleep",
		  name = "Thread.sleep should not be used",
		  priority = Priority.CRITICAL
		)
		public class NoThreadSleepRule extends IssuableSubscriptionVisitor {

			@Override
			public List<Tree.Kind> nodesToVisit() {
				return Collections.singletonList(Tree.Kind.METHOD_INVOCATION);
			}

			@Override
			public void visitNode(Tree tree) {
				MethodInvocationTree mit = (MethodInvocationTree) tree;
				if (mit.symbol().name().equals("sleep")) {
					reportIssue(mit, "Use Selenium Wait Strategy instead of Thread.sleep");
				}
			}
		}


			✔ Enterprise-grade
			❌ Requires Java AST knowledge
			❌ Maintenance-heavy

🔹 B) Rule Templates (Limited Customization)

Sonar allows rule templates, where you:

Clone an existing rule
Change thresholds / regex
Change severity

Example:

Maximum allowed method length
Cyclomatic complexity threshold
Regex-based naming rules

		✔ Easy
		❌ Limited flexibility

4️⃣ Option 3 – Import Rules from External Tools

SonarQube integrates with:

Tool					Purpose
Checkstyle				Coding standards
PMD						Code quality
SpotBugs				Runtime bug detection

Example:

Custom Checkstyle XML for automation standards

Enforce page object naming
Restrict static variables
Sonar imports violations as issues.

5️⃣ Automation Framework–Specific Rules (Real World)

In Selenium / Cucumber frameworks, teams often add rules like:

Rule												Reason
No Thread.sleep										Flaky tests
No static WebDriver									Parallel issues
Page classes must not contain assertions			SRP
Step definitions must not use WebDriver directly	Architecture
No hard-coded test data								Maintainability


6️⃣ What You CANNOT Do

❌ Write rules directly in UI (except templates)
❌ Detect runtime flakiness
❌ Detect whether a test caught a bug
❌ Add business logic rules

Sonar = static analysis only

7️⃣ Interview-Ready Answer (Senior Level)

“Yes, SonarQube supports adding rules in multiple ways. Most teams customize Quality Profiles by enabling or disabling built-in rules. For advanced needs, custom rules can be implemented via Sonar plugins using the Java analyzer API, or by integrating external tools like Checkstyle or PMD. Rule templates allow limited customization without coding. However, SonarQube only supports static analysis and cannot detect runtime behavior.”

8️⃣ One-Liner for Interviews

“You don’t usually add new rules — you activate, tune, or extend existing ones.”




--------------------------------------------------------------------------------------------------------------------------------------

When NOT to Write Custom SonarQube Rules
🔴 Rule #1: If an Existing Rule Already Covers It
❌ Don’t write a custom rule when:

Sonar already has a built-in rule Or the issue can be caught via:
Quality Profile tuning
Rule severity changes
Threshold adjustment

Example (Automation)

❌ Writing a rule for:

Long methods
High cyclomatic complexity
Duplicate code
Unused variables

✔ These already exist

Interview line:
“Custom rules are unnecessary if built-in rules can be configured to enforce the same behavior.”

🔴 Rule #2: If the Issue Is Runtime / Behavioral

SonarQube performs static analysis only.

❌ Don’t write rules for:

Flaky tests
Timing issues
API instability
Selenium synchronization problems
Network failures
Environment-dependent failures

Example

❌ “Detect flaky Selenium tests”
❌ “Check if explicit wait is long enough”

✔ These require execution data, not source code

Interview line:
“Static analysis cannot reason about runtime behavior.”

🔴 Rule #3: If the Rule Is Project-Specific
❌ Don’t write custom rules when:

Only one project needs it
Temporary workaround
Team preference, not organization standard

Example

❌ “All tests must use XYZ naming” (only for one repo)
❌ “Use Chrome only” (environment constraint)

✔ Use code reviews or linters instead

Interview line:
“Custom rules should represent organization-wide standards, not local conventions.”

🔴 Rule #4: If Maintenance Cost > Benefit

Custom rules:

Break with Sonar upgrades
Need AST expertise
Require plugin maintenance
Need documentation + training

❌ Don’t write rules when:

Rule logic is complex
Analyzer version changes often
Team lacks Sonar plugin expertise

Example

❌ Custom rule checking Page Object + Step Definition interaction

✔ Better solved via:

Architecture tests
Code reviews
Framework design

Interview line:
“If the rule costs more to maintain than the defects it prevents, it’s not worth it.”

🔴 Rule #5: If the Problem Is Architectural

Sonar rules work per file, not system-wide architecture.

❌ Don’t write rules for:

Layer violations
Incorrect framework usage
Wrong dependency direction

Example

❌ “Step Definitions must not use WebDriver”
❌ “Page Objects must not call APIs”

✔ Use:

ArchUnit

Design reviews

Framework enforcement

🔴 Rule #6: If Rule Depends on Business Logic
❌ Don’t write rules when:

Logic depends on domain knowledge
Business flows
Test data semantics

Example

❌ “Registration test must validate OTP”
❌ “API response must contain correct discount logic”

✔ These belong in tests, not static analysis

✅ When You SHOULD Write Custom Rules (Contrast)

Write Rule When					Reason
Organization-wide standard		Consistency
Repeated violations				High ROI
Simple AST logic				Low maintenance
Security/compliance				Mandatory
Automation anti-patterns		Framework health


Example (Good Custom Rule)

✔ No Thread.sleep()
✔ No static WebDriver
✔ No assertions in Page Objects
✔ No hard-coded credentials

🎯 Interview-Ready Summary (Perfect Answer)

“Custom SonarQube rules should only be written when existing rules cannot enforce an organization-wide standard and the benefit outweighs the long-term maintenance cost. They should not be used for runtime behavior, project-specific conventions, business logic, or architectural enforcement. In most cases, tuning existing rules or using complementary tools like Checkstyle or ArchUnit is the better approach.”
