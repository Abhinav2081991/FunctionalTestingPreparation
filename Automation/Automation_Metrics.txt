📊 Automation Metrics Dashboard — What Actually Matters

Think in four layers:

1️⃣ Pipeline Health
2️⃣ Test Quality
3️⃣ Framework Health
4️⃣ Business Impact

Not just test counts.

🟢 1. CI Pipeline Health Metrics

These answer: “Can I trust today’s build?”

✅ Build Pass Rate (Daily / Weekly)

Formula

Successful Builds / Total Builds

Why it matters

Shows CI stability. If this drops → flaky tests or infra issues.

Target

👉 90%+

✅ Mean Time to Detect Failure (MTTD)

Time between:

Code merged → Test failure detected


Lower is better.

✅ Mean Time to Repair (MTTR)

How long it takes to fix broken automation.

Critical for leadership.

✅ Pipeline Duration Trend

Track:

Smoke duration

Regression duration

Sudden spikes = environment or framework regression.

Dashboard Visualization

Line graph: Build duration over time

Bar chart: Failed vs Passed builds

🧪 2. Test Quality Metrics (Most Important)

These answer: “Are tests reliable?”

⭐ Flakiness Index (Gold Metric)
Definition

Tests that fail once but pass on rerun.

Flaky Tests / Total Failed Tests

Why

High flakiness = automation debt.

Target

< 5%

✅ Failure Root Cause Distribution

Pie chart:

Test issue
Application bug
Environment
Data issue

This tells you where engineering time is bleeding.

✅ Assertion Density

Average assertions per test.

Too low → weak validation
Too high → brittle tests

Sweet spot: 3–6

✅ Retry Rate

If retry rate increases → framework instability.

🧱 3. Framework Health Metrics

These answer: “Is our automation maintainable?”

✅ Page Object Size

Track:

Methods per Page
Lines of code
Alert when:

Page > 300 LOC

Indicates God Objects.

✅ JSExecutor Usage

Count how many tests use JS.

Rising trend = poor locator strategy.

✅ Thread.sleep Count

Should be:

ZERO


Track via static analysis.

✅ Duplicate Locator Ratio

Same XPath/CSS repeated across pages.

High value = weak abstraction.

🌐 4. API Automation Metrics (REST Assured)

If you’re using REST Assured with Java:

✅ API Pass Rate

Baseline confidence for backend stability.

✅ Contract Failures

Track schema mismatches.

Great indicator of breaking changes.

✅ Response Time Trends

90th percentile response times per endpoint.

Early performance signal.

🧠 5. Coverage Metrics (Use Carefully)

Coverage ≠ Quality, but still useful.

✅ Feature Coverage
Automated Scenarios / Total Scenarios

✅ API vs UI Split

Target mature teams:

60% API

30% UI

10% E2E

If UI > API → slow feedback.

🐞 6. Defect Correlation Metrics (from Jira)

These answer: “Is automation preventing bugs?”

✅ Escaped Defects

Production bugs that automation should have caught.

Track monthly.

✅ Automation Catch Rate
Bugs caught by automation / Total bugs


Leadership LOVES this.

🚫 Real Examples of BAD Automation Metrics
1️⃣ “Total Number of Automated Test Cases”
Example dashboard tile

📊 Total Tests: 3,842

Why teams love it

It looks impressive.
Management likes big numbers.

Why it’s bad

It tells you NOTHING about:

❌ stability
❌ quality
❌ coverage
❌ usefulness
❌ maintenance cost

You can have:

3,800 flaky tests

3,800 duplicated tests

3,800 tests asserting nothing

Still shows “3,800”.

Pure vanity.

✅ Replace with

Pass rate trend

Flakiness index

Feature coverage

2️⃣ “Lines of Automation Code Written”

Yes — this exists.

Why it’s bad

More lines usually means:

❌ more duplication
❌ worse design
❌ bigger maintenance cost

Great automation is smaller, not larger.

Real outcome

Teams start:

avoiding refactoring

writing verbose code

copy-pasting Page Objects

Because they’re “rewarded” for LOC.

Disaster.

✅ Replace with

Duplicate locator ratio

Page Object size

Reuse percentage

3️⃣ “Number of Selenium Scripts Executed”
Why it’s misleading

Running 1,000 scripts doesn’t mean:

they asserted anything

they covered risk

they found defects

Often it just means:

UI tests being spammed in CI.

Common smell

Huge regression packs with:

overlapping scenarios

minor UI variations

no business value

✅ Replace with

Smoke pass rate

Business flow coverage

API vs UI ratio

4️⃣ “Automation Coverage = Test Count / Manual Test Count”

Classic spreadsheet metric.

Why it’s wrong

Manual tests ≠ automation candidates.

Some manual tests:

exploratory

UX

visual

one-time

Should NEVER be automated.

So this metric always lies.

✅ Replace with

Feature-based coverage:

Automated business flows / total business flows

5️⃣ “Build Passed Today = Green”

Single-day pass status.

Why it’s dangerous

Today might be green because:

retries masked failures

flaky tests got lucky

nothing changed

No trend. No history.

✅ Replace with

7/14/30 day pass trend

Retry rate

MTTR

6️⃣ “Retry Success Rate”

This one is subtle.

Teams proudly show:

85% failures fixed by retry

Reality

That means:

👉 tests are flaky
👉 framework unstable
👉 infra unreliable

Retry success is NOT success.

It’s technical debt.

✅ Replace with

Flakiness Index:

Tests failing once but passing later / total failures

7️⃣ “Number of Bugs Found by Automation”

Sounds good.

Why it’s misleading

Automation doesn’t find bugs.

It detects regressions you already modeled.

High count could mean:

unstable product

weak unit tests

late automation

Low count might mean:

good development quality

Context matters.

✅ Replace with

Automation Catch Rate:

Bugs caught pre-prod / total bugs


and

Escaped defects.

8️⃣ “Percentage of UI Automation”

Seen in many orgs:

80% UI automation 🎉

Why it’s actually bad

UI tests are:

slow

brittle

expensive

High UI % usually means:

❌ poor API strategy
❌ late testing
❌ slow feedback

Mature teams target:

~60% API

~30% UI

~10% E2E

9️⃣ “Average Test Execution Time”

Without context.

Why it’s meaningless

Is that:

smoke?

regression?

API?

UI?

Mixing all gives nonsense.

✅ Replace with

Separate:

Smoke duration
Regression duration
API suite duration
Trend each independently.

🔟 “Number of PRs with Automation”

This pushes engineers to add:

trivial tests

meaningless assertions

just to tick a box.

Result

Low-quality automation everywhere.

✅ Replace with

PR Quality Gates:

No sleeps

Proper waits

Assertions present

Design respected

✅ Examples of GOOD Automation Metrics (with Formulas)
1️⃣ Automation Stability (Pass Rate Trend)
📐 Formula
(Total Passed Tests / Total Executed Tests) × 100

Example
920 passed / 1000 executed = 92%

Why this matters

This tells you:

✔ framework health
✔ infra reliability
✔ test robustness

But only when viewed as a trend over time (7 / 14 / 30 days).

Healthy Range

Smoke: >95%

Regression: >90%

Anything below → investigate.

Leadership Insight

Never look at single-day pass rate. Always look at trend slope.

2️⃣ Flakiness Index (Most Important Metric)
📐 Formula
(Number of tests that failed once but passed on retry / Total failed tests) × 100

Example
30 flaky / 100 failures = 30%

Why this matters

This exposes:

❌ timing issues
❌ unstable locators
❌ async UI problems
❌ infra noise

Retries hide this.
This metric reveals it.

Healthy Range

Elite teams: <5%

Acceptable: <10%

15% → automation credibility collapsing

Real-world use

Top teams block releases if flakiness crosses threshold.

3️⃣ Mean Time To Repair (MTTR) – Automation Failures
📐 Formula
Sum of fix durations / Number of failures

Example

Failures took: 2h + 3h + 1h = 6h

6h / 3 = 2h MTTR

Why this matters

Measures:

✔ framework maintainability
✔ team responsiveness
✔ code clarity

Low MTTR = clean architecture.

Healthy Range

API tests: <30 min

UI tests: <2 hours

4️⃣ Escaped Defect Rate
📐 Formula
Production bugs / Total bugs

or

Prod bugs / (QA + UAT + Prod bugs)

Why this matters

This directly reflects test effectiveness, not volume.

Healthy Range

<5–10%

Higher means:

❌ missing coverage
❌ weak assertions
❌ poor risk modeling

5️⃣ Automation Catch Rate
📐 Formula
Bugs caught by automation / Total regression bugs

Why this matters

Shows how much regression risk automation is absorbing.

Mature teams

Target: 60–70%

If low:

tests too UI-focused

API layer missing

assertions weak

6️⃣ API vs UI Automation Ratio
📐 Formula
API Tests / Total Automated Tests

and

UI Tests / Total Automated Tests

Healthy Benchmark

Approx:

60% API
30% UI
10% E2E

Why this matters

High UI % = slow CI + fragile pipelines.
Senior teams push left into API.

7️⃣ Failure Root Cause Distribution
Categories

Automation defect
Product defect
Environment
Test data

📐 Formula
Failures per category / Total failures

Why this matters
Shows where engineering investment is needed.

Example:

40% automation failures → framework debt
30% env → DevOps issue


Instant action plan.

8️⃣ Page Object Complexity Metric
Example Formula
Average methods per Page Object


or

Lines per Page Object
Healthy Range
Methods: <10–12

Lines: <300

Large pages = god objects.

9️⃣ Regression Duration Trend
📐 Formula
Total regression runtime per build


Tracked over time.

Why this matters

Regression growing week over week means:

❌ uncontrolled test growth
❌ missing suite partitioning

Elite teams keep this flat.

🔟 PR Automation Quality Gate

Instead of counting tests.

Track:

PRs passing automation quality checks / total PRs


Checks include:

✔ no Thread.sleep
✔ waits centralized
✔ assertions present
✔ page objects respected
