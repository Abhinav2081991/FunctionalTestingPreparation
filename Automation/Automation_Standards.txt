🏗 1. Framework Architecture Standards
Mandatory Layering
Tests / Step Definitions
↓
Business Flow Layer
↓
Page Objects / API Clients
↓
Driver / HTTP Factories
↓
Utilities
↓
Config

Rules

❌ No Selenium code in Tests
❌ No Assertions in Page Objects
❌ No API calls inside UI Pages
❌ No WebDriver static variables

Example

Bad:

driver.findElement(By.id("login")).click();


Good:

loginPage.login(user);

🧱 2. Page Object Standards (UI)
Each Page Must:

✅ Represent ONE page
✅ Expose behaviors, not elements
✅ Have <12 public methods
✅ Return next page objects

Example
public DashboardPage login(User user){
   enterUsername(user.getName());
   enterPassword(user.getPassword());
   clickLogin();
   return new DashboardPage();
}

Naming Convention
LoginPage
DashboardPage
SettingsPage


Never:

LoginPageTest
LoginPageHelper

⏳ 3. Wait Strategy Standards (Critical)
Rule

👉 Zero Thread.sleep

Centralized waits only:

WaitFactory.waitFor(locator, WaitType.CLICKABLE);

Never:
Thread.sleep(3000);


Why?

Hard waits = flaky pipelines.

🧪 4. BDD Standards (Cucumber)
Feature files must be BUSINESS readable

Bad:

Given user clicks login button


Good:

Given user logs into application

Step definitions must call business flows
@Given("user logs in")
public void login(){
   loginFlow.perform();
}


Not Selenium commands.

Scenarios must be independent

❌ No scenario chaining
❌ No shared state

🔁 5. TDD Standards (API + Logic)
Test first:
@Test
public void shouldCreateOrder(){
   assertEquals(201, createOrder());
}


Then implementation.

REST Assured:

Create reusable API clients:

OrderClient.createOrder(payload);


Never inline HTTP calls in tests.

🌐 6. REST Assured Standards
API Layer Structure
client/
   OrderClient.java
model/
   OrderRequest.java
   OrderResponse.java

Example
public static Response createOrder(OrderRequest req){
   return given()
          .body(req)
          .post("/order");
}

Validation separated:
OrderValidator.validateCreated(response);

🧵 7. Parallel Execution Standards
Mandatory:
ThreadLocal<WebDriver>

Never:
static WebDriver driver;


Reason:

Parallel runs + static driver = corrupted sessions.

📦 8. Test Data Standards
Immutable test data objects
public record User(String name, String password){}

Data Sources:

✅ API seeding
✅ JSON fixtures
✅ Builders

Never Excel.

Every failure must capture:

✅ Screenshot
✅ DOM dump

Automatically.

Tokens

✅ Automation Coding Standards Checklist (Senior / Lead Level)

Use this as a pre-merge gate.

🏗 Architecture & Design

Tests do NOT directly use WebDriver / REST Assured

Clear layering exists:

Tests / Steps
→ Business Flows
→ Page Objects / API Clients
→ Driver / HTTP Factory
→ Utils / Config


No Selenium logic inside Step Definitions
No API calls inside Page Objects
No assertions inside Page Objects
Each Page Object represents ONE screen / component
Page Objects expose behavior, not elements

🧱 Page Object Standards (UI)

Public methods ≤ ~10–12 per page (no God Pages)
Methods return next Page Object where applicable
Locators are private and centralized
No hardcoded waits or sleeps
No findElement calls inside tests
Example expectation:
dashboardPage.createOrder(order);


Not:

driver.findElement(...).click();

⏳ Wait Strategy

❌ No Thread.sleep

Explicit waits are centralized (WaitFactory / utility)

No implicit + explicit wait mixing

JSExecutor is NOT used for normal clicks/typing

SPA apps handled with proper DOM readiness checks

🧪 BDD Standards (if using Cucumber)

Feature files describe BUSINESS behavior (not UI actions)

Steps call business flows, not Selenium

Scenarios are independent (no chaining)

Background used only for true common setup

Tags used for smoke / regression / api / ui separation

Bad:

Given user clicks login button


Good:

Given user logs into application

🌐 REST Assured / API Standards

API calls abstracted into Client classes

Request/Response models separated (POJOs / records)

Validation logic separated from API execution

No inline JSON strings inside tests

Base URI, headers, auth come from config

Expected structure:

client/
model/
validator/
tests/

🧵 Parallel Execution & Thread Safety

WebDriver managed via ThreadLocal

No static WebDriver

No shared mutable test data

Parallel execution tested locally

📦 Test Data

Immutable test data objects

Data created via API where possible

No Excel dependency

No hardcoded credentials

Environment-specific values externalized

🚨 Failure Handling

Every failure must automatically capture:

Screenshot

Browser console logs

DOM dump (optional)

API request/response (for API tests)

🔐 Config & Secrets

No credentials in code

URLs come from env/config

Tokens handled via pipeline variables or vault

Supports multiple environments (qa / stage / prod)

📊 Reporting

Reports show:

Scenario name

Environment

Browser

Failure screenshot

CI artifacts uploaded automatically

🧠 Java Code Quality

No large utility dumping grounds

Streams used sensibly (no parallel streams with WebDriver)

Optionals handled safely

No empty catch blocks

Meaningful method and variable names
