🔍 What a true Singleton is

A classic Singleton has all 3 guarantees:

Exactly one instance of the class
Private constructor
Controlled access to that single instance

Example (textbook):

public class ClassicSingleton {

    private static final ClassicSingleton INSTANCE = new ClassicSingleton();

    private ClassicSingleton() {}

    public static ClassicSingleton getInstance() {
        return INSTANCE;
    }
}

🧪 Now analyze your DriverManager
public final class DriverManager {

    public static ThreadLocal<WebDriver> threadLocalDriver = new ThreadLocal<>();

    private DriverManager() {}

    public static WebDriver getDriver() {
        return threadLocalDriver.get();
    }

    public static void setDriver(WebDriver driver) {
        threadLocalDriver.set(driver);
    }

    public static void unload() {
        threadLocalDriver.remove();
    }
}

✅ What makes it look like a Singleton

✔ Private constructor
✔ Final class
✔ Global access via static methods
✔ Single “manager” role

➡ This matches the intent of Singleton (centralized access)

❌ Why it is NOT a true Singleton
🚫 There is no instance of DriverManager

Everything is static
No getInstance()
No object lifecycle
DriverManager dm = new DriverManager(); // impossible, but also unnecessary


So technically:

No object → no singleton instance

🧠 What this pattern ACTUALLY is
✅ Singleton-style Utility Class

Sometimes called:

Static Holder Pattern

Driver Context

Thread-Scoped Resource Manager

👉 Very common in Selenium frameworks
👉 Purpose is driver lifecycle control, not object instantiation

🧵 Why ThreadLocal changes the game
public static ThreadLocal<WebDriver> threadLocalDriver = new ThreadLocal<>();


This means:

One WebDriver per thread

But still ONE global access point

So effectively:

Singleton per thread, not per JVM
This is exactly what you want in:
Parallel execution
Selenium Grid
Jenkins multi-node runs

🎯 Interview-Perfect Answer

“This is not a classical Singleton because no instance is created. It’s a static utility class that provides centralized, thread-safe access to WebDriver using ThreadLocal. Conceptually it behaves like a Singleton per thread, which is ideal for parallel Selenium execution.”

public final class DriverManager {

    private static final DriverManager INSTANCE = new DriverManager();
    private ThreadLocal<WebDriver> driver = new ThreadLocal<>();

    private DriverManager() {}

    public static DriverManager getInstance() {
        return INSTANCE;
    }

    public WebDriver getDriver() {
        return driver.get();
    }

    public void setDriver(WebDriver webDriver) {
        driver.set(webDriver);
    }
