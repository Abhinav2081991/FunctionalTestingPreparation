Common Selenium Patterns That Fail in CI/CD (But Work Locally)
1️⃣ Missing Explicit Waits (Timing Dependency)
❌ Works Locally
driver.findElement(By.id("submit")).click();

💥 Fails in CI

Slower CPU
No GPU
Headless rendering delay

✅ Fix
WebElement submit = new WebDriverWait(driver, Duration.ofSeconds(10))
        .until(ExpectedConditions.elementToBeClickable(By.id("submit")));
submit.click();


📌 CI is slower, not broken

2️⃣ Hardcoded Thread.sleep()
❌ Works Locally
Thread.sleep(2000);
driver.findElement(By.id("save")).click();

💥 Fails in CI

Animation takes longer
Network latency
Unpredictable load

✅ Fix
wait.until(ExpectedConditions.invisibilityOfElementLocated(By.id("loader")));
driver.findElement(By.id("save")).click();

3️⃣ Headless Viewport Issues
❌ Works Locally
driver.findElement(By.id("login")).click();

💥 Fails in CI (Headless)

Element is off-screen
Not interactable

✅ Fix
options.addArguments("--window-size=1920,1080");

((JavascriptExecutor) driver)
        .executeScript("arguments[0].scrollIntoView(true);", element);
element.click();

4️⃣ ElementClickInterceptedException (Overlays)
❌ Works Locally
driver.findElement(By.xpath("//button[text()='Submit']")).click();

💥 Fails in CI

Cookie banner
Loader overlay
A/B testing banners

✅ Fix
wait.until(ExpectedConditions.elementToBeClickable(button)).click();


Or JS fallback (last resort).

5️⃣ Absolute XPaths (Fragile Locators)
❌ Works Locally
By.xpath("/html/body/div[2]/div[3]/button[1]");

💥 Fails in CI

DOM rendered slightly differently

Responsive layout

✅ Fix
By.xpath("//button[normalize-space()='Submit']");

6️⃣ File Upload Using Relative Paths
❌ Works Locally
fileInput.sendKeys("src/test/resources/file.pdf");

💥 Fails in CI

Different workspace path

✅ Fix
Path filePath = Paths.get("src/test/resources/file.pdf").toAbsolutePath();
fileInput.sendKeys(filePath.toString());

7️⃣ Assumption That Tests Run Sequentially
❌ Works Locally
static WebDriver driver;

💥 Fails in CI

Parallel execution
Thread collision

✅ Fix
private static ThreadLocal<WebDriver> driver = new ThreadLocal<>();

8️⃣ Using Browser Cache / Cookies
❌ Works Locally
driver.get("https://app.example.com/dashboard");

💥 Fails in CI

Clean environment

No cached login

✅ Fix
loginPage.login(user, password);


📌 CI = fresh machine every time

9️⃣ Hover-Based Actions
❌ Works Locally
actions.moveToElement(menu).perform();
submenu.click();

💥 Fails in CI

Headless doesn’t hover well

✅ Fix
wait.until(ExpectedConditions.visibilityOf(submenu)).click();


or JS-trigger hover.

🔟 JavaScript Execution Timing
❌ Works Locally
((JavascriptExecutor) driver)
    .executeScript("document.querySelector('#btn').click()");

💥 Fails in CI

DOM not fully loaded

✅ Fix
wait.until(webDriver ->
    ((JavascriptExecutor) webDriver)
        .executeScript("return document.readyState").equals("complete"));

1️⃣1️⃣ Font / CSS Differences
❌ Works Locally
assertEquals("Submit", button.getText());

💥 Fails in CI

Different fonts / trimming

Upper/lower case rendering

✅ Fix
assertTrue(button.getText().trim().equalsIgnoreCase("Submit"));

1️⃣2️⃣ Dependency on Screen Resolution
❌ Works Locally
assertTrue(element.isDisplayed());

💥 Fails in CI

Responsive UI hides element

✅ Fix
options.addArguments("--window-size=1920,1080");

🧠 Golden Rule for CI/CD Selenium

If it works locally but fails in CI, your test is timing-dependent, not the environment.

✅ CI-Safe Selenium Checklist (Print This)

✔ Explicit waits everywhere
✔ No Thread.sleep
✔ Stable locators
✔ Headless viewport set
✔ No shared static state
✔ Retry + JS fallback only as last resort
✔ Screenshots + logs on failure

Selenium CI Failure Debugging Playbook

(Local Pass ❌ | CI Fail 💥)

🔴 STEP 0: Classify the Failure (Don’t Guess)

First question you must answer:

Symptom	Category
Element not found / not clickable	Timing / UI sync
Works locally only					Headless / resolution
Random failures						Flakiness / async
Only in parallel					Thread safety
Passes on rerun						Retry masking
Screenshot shows wrong page			Navigation / auth

📌 Never debug blindly — classify first.

🔍 STEP 1: Always Capture Evidence
✅ Mandatory on Every CI Failure
@AfterStep
public void afterStep(Scenario scenario) {
    if (scenario.isFailed()) {
        takeScreenshot();
        savePageSource();
        logBrowserConsole();
    }
}

📸 Screenshot
byte[] screenshot = ((TakesScreenshot) driver)
        .getScreenshotAs(OutputType.BYTES);
scenario.attach(screenshot, "image/png", "Failure Screenshot");

📄 Page Source
Files.write(
    Paths.get("target/pagesource.html"),
    driver.getPageSource().getBytes()
);

🟠 STEP 2: Reproduce CI Locally (Critical)
🔁 Run exact CI config locally
mvn clean test \
-Dheadless=true \
-Dbrowser=CHROME \
-Dexecution=LOCAL


Same:

Java version
Chrome version
Maven profile
Headless flags

📌 If you can’t reproduce locally, you’re blind.

🟡 STEP 3: Headless-Specific Fixes
❌ Common CI Breaker
driver.findElement(By.id("submit")).click();

✅ CI-Safe Pattern
options.addArguments("--window-size=1920,1080");
wait.until(ExpectedConditions.elementToBeClickable(submit)).click();

🧠 Debug Trick

Run headed in CI once:

--headless=false


If it passes → viewport issue confirmed

🔵 STEP 4: Validate Timing & Async Issues
❌ Smells Bad
Thread.sleep(2000);

✅ Smells Good
wait.until(ExpectedConditions.invisibilityOf(loader));
wait.until(ExpectedConditions.elementToBeClickable(button));

🟣 STEP 5: Detect Overlay / Interception
Symptom
ElementClickInterceptedException

🔍 Debug Overlay
((JavascriptExecutor) driver)
    .executeScript("document.body.style.border='5px solid red'");

✅ Fix
wait.until(ExpectedConditions.elementToBeClickable(element));


JS click only as last resort.

🟤 STEP 6: Parallel Execution Bugs
❌ Local Single Thread
static WebDriver driver;

💥 CI Parallel Failure

Null driver

Wrong browser

Random failures

✅ Fix
private static ThreadLocal<WebDriver> driver = new ThreadLocal<>();


📌 If failure disappears when parallelism = 1 → thread safety bug

⚫ STEP 7: Environment Validation

Add this at test start:

System.out.println("Java: " + System.getProperty("java.version"));
System.out.println("OS: " + System.getProperty("os.name"));
System.out.println("Browser: " + capabilities.getBrowserVersion());


CI failures often hide version mismatch issues.

🔥 STEP 8: DOM Reality Check
❌ Assumption
assertEquals("Submit", button.getText());

💥 CI Reality

Extra spaces
Uppercase
Localization

✅ Fix
assertTrue(button.getText().trim().equalsIgnoreCase("submit"));

🧯 STEP 9: Retry (But Be Honest)
❌ Bad
retry(5); // hides bugs

✅ Good
retry(2); // logs + screenshots + metrics


📌 Retries without root cause = technical debt

🧠 STEP 10: CI-Only Debug Hooks (Hidden Gold)
Log browser console
driver.manage().logs().get(LogType.BROWSER)
      .forEach(log -> System.out.println(log.getMessage()));

Dump active overlays
executeScript(
 "return document.querySelectorAll('div[style*=\"position: fixed\"]').length");

🧪 CI Debug Decision Tree
Fails only in CI?
  ├── Yes → Headless / Timing / Resolution
  ├── Random → Async / Parallel / Overlay
  ├── Only on first run → Auth / Cache
  └── Only in parallel → ThreadLocal issue

✔ Passes 10+ CI runs
✔ Passes with parallel = 5
✔ Passes headless + headed
✔ No retries needed
✔ Failure screenshots are clean
