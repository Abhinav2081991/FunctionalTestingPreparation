Steps to Add Global Credentials via the Web UI
Log in to your Jenkins instance with a user account that has the necessary permissions (e.g., administrator).
From the Jenkins Dashboard, navigate to Manage Jenkins.
Under the "Security" heading, click on Manage Credentials.
Under "Stores scoped to Jenkins", click on System.
Select the Global credentials (unrestricted) link to access the default domain.
In the left-hand menu, click Add Credentials.
In the configuration form, use the Kind dropdown field to select the type of credential you wish to add. Common types include:
Secret text (for API tokens or generic secrets).
Username and password (for basic authentication).
SSH Username with private key.
Secret file (for configuration files, certificates, etc.).
Certificate (PKCS#12 file and optional password).
Choose the Scope (usually "Global" for general use).
Fill in the specific fields for your selected credential type (e.g., Username, Password, Private Key).
Specify a unique and meaningful ID for the credential, which you will use to reference it in Pipeline scripts or job configurations.
Add an optional Description.
Click OK to save the new credential. 



Secure Storage of Keystore, Certificates, Tokens & Truststore in Jenkins

🧭 First: What goes where? (Big Picture)

Secret Type	Store In Jenkins As	Why
API Tokens / JWT	Secret Text	Simple string
Username / Password	Username + Password	Native support
Keystore (.jks, .p12)	Secret File	Binary + password
Truststore (.jks)	Secret File	Binary
OAuth Client Secret	Secret Text	Rotation-friendly
Certificates (.pem, .crt)	Secret File	Binary

✔ Never commit these into Git
✔ Never hardcode in code or pom.xml

2️⃣ Fetching Secrets in Jenkinsfile (Most Important)
🔹 A. Fetch API Token (Secret Text)
withCredentials([string(credentialsId: 'API_TOKEN', variable: 'TOKEN')]) {
    sh '''
        mvn test -Dapi.token=$TOKEN
    '''
}

👉 Use in Rest Assured
String token = System.getProperty("api.token");

given()
    .auth()
    .oauth2(token)
.when()
    .get("/users");

🔹 B. Fetch Username & Password
withCredentials([usernamePassword(
    credentialsId: 'BASIC_AUTH',
    usernameVariable: 'USER',
    passwordVariable: 'PASS'
)]) {
    sh '''
        mvn test -Duser=$USER -Dpass=$PASS
    '''
}

🔹 C. Fetch Keystore / Truststore (Secret File)


✅ Best practice: Jenkins copies file to workspace temp path
withCredentials([file(credentialsId: 'CLIENT_KEYSTORE', variable: 'KEYSTORE_FILE')]) {
    sh '''
        mvn test -Dkeystore.path=$KEYSTORE_FILE
    '''
}

👉 Rest Assured usage
String keystorePath = System.getProperty("keystore.path");
String keystorePassword = System.getenv("KEYSTORE_PASS");

given()
    .auth()
    .certificate(keystorePath, keystorePassword)
.when()
    .get("/secure-api");

🔹 D. Truststore Fetching
withCredentials([file(credentialsId: 'TRUSTSTORE_JKS', variable: 'TRUSTSTORE_FILE')]) {
    sh '''
        mvn test -Dtruststore.path=$TRUSTSTORE_FILE
    '''
}

👉 JVM SSL properties (recommended)
-Djavax.net.ssl.trustStore=${TRUSTSTORE_FILE}
-Djavax.net.ssl.trustStorePassword=changeit

3️⃣ Recommended Jenkins Secret Layout (Enterprise)
Credentials
├── API_TOKEN (Secret Text)
├── OAUTH_CLIENT_SECRET (Secret Text)
├── BASIC_AUTH (Username/Password)
├── CLIENT_KEYSTORE (Secret File)
├── TRUSTSTORE_JKS (Secret File)
├── KEYSTORE_PASS (Secret Text)
├── TRUSTSTORE_PASS (Secret Text)

4️⃣ CI/CD-Safe Jenkinsfile Example (Complete)
pipeline {
    agent any

    stages {
        stage('Run Secure API Tests') {
            steps {
                withCredentials([
                    string(credentialsId: 'API_TOKEN', variable: 'TOKEN'),
                    file(credentialsId: 'CLIENT_KEYSTORE', variable: 'KEYSTORE'),
                    string(credentialsId: 'KEYSTORE_PASS', variable: 'KPASS')
                ]) {
                    sh '''
                        mvn test \
                        -Dapi.token=$TOKEN \
                        -Dkeystore.path=$KEYSTORE \
                        -Dkeystore.password=$KPASS
                    '''
                }
            }
        }
    }
}

5️⃣ Why NOT Store in pom.xml or Git?

❌ pom.xml → visible in logs
❌ Git → security breach
❌ Jenkins env vars globally → leaked to other jobs

✔ Jenkins Credentials → masked in logs
✔ Scoped per job/folder
✔ Rotatable without code change

6️⃣ Handling Certificates in Docker / Kubernetes Jenkins
🔹 Mount certs at runtime
volumeMounts:
- name: keystore
  mountPath: /certs

-Dkeystore.path=/certs/client.p12

7️⃣ Common CI Failures & Fixes
Error	Root Cause	Fix
PKIX path building failed	Missing truststore	Add truststore
File not found	Wrong credentialsId	Check Jenkins ID
SSLHandshakeException	Cert mismatch	Correct keystore
Token expired	Static token	Dynamic token job
🧠 Interview-Ready Summary

Keystores, truststores, certificates, and tokens should be stored securely in Jenkins Credentials — using Secret Text for tokens and Secret File for binary artifacts. They are injected at runtime using withCredentials, passed as JVM or Maven properties, and never committed to source control.
